# Update Tech Stack Skill

Scan the project to detect the technology stack and generate or update `<basedir>/tech-stack.md`. This living document is referenced by CLAUDE.md and other skills to understand project conventions.

## Step 0: Read config and verify project

Read the shared config reference:
```
~/.claude/skills/shared/config.md
```

Resolve the basedir from `.btrs-config.json` (default: `.local`). Verify the project is initialized by checking that the basedir directory exists.

## Step 1: Scan for stack indicators

Systematically check for the following. Read each file that exists — don't just check for presence:

### Package management & runtime
- `package.json` — Read for name, engines, scripts, dependencies, devDependencies
- `pnpm-lock.yaml` / `package-lock.json` / `yarn.lock` / `bun.lockb` — Detect package manager
- `deno.json` / `deno.lock` — Deno
- `Cargo.toml` — Rust
- `go.mod` — Go
- `requirements.txt` / `pyproject.toml` / `Pipfile` — Python
- `Gemfile` — Ruby
- `.python-version` / `.node-version` / `.nvmrc` / `.tool-versions` — Runtime versions

### Frameworks & meta-frameworks
- `svelte.config.js/ts` — SvelteKit
- `next.config.js/ts/mjs` — Next.js
- `nuxt.config.ts/js` — Nuxt
- `vite.config.js/ts` — Vite
- `astro.config.mjs` — Astro
- `angular.json` — Angular
- `remix.config.js` — Remix
- `nest-cli.json` — NestJS

### Styling
- `tailwind.config.js/ts` — Tailwind CSS
- `postcss.config.js/cjs` — PostCSS
- `*.css` / `*.scss` / `*.less` presence and conventions
- CSS variable files / design token files

### Database & ORM
- `prisma/schema.prisma` — Prisma
- `drizzle.config.ts` — Drizzle
- `knexfile.js` — Knex
- `ormconfig.json` / `typeorm` in deps — TypeORM
- `docker-compose.yml` — Check for database services (postgres, mysql, redis, mongo)

### Testing
- `vitest.config.ts` / `jest.config.js` / `playwright.config.ts` / `cypress.config.ts`
- Test directory structure (`__tests__/`, `*.test.*`, `*.spec.*`)

### CI/CD & deployment
- `.github/workflows/` — GitHub Actions
- `.gitlab-ci.yml` — GitLab CI
- `Dockerfile` / `docker-compose.yml`
- `Jenkinsfile` / `Makefile` / `Taskfile.yml`
- `vercel.json` / `netlify.toml` / `fly.toml`

### Code quality & tooling
- `.eslintrc*` / `eslint.config.*` — ESLint
- `.prettierrc*` / `prettier.config.*` — Prettier
- `tsconfig.json` — TypeScript config (read for strict mode, paths, target)
- `.editorconfig`
- `biome.json` — Biome

### Project conventions (detect from code)
- Check `src/` structure for patterns (routes, components, lib, utils, stores, etc.)
- Check for component naming conventions (PascalCase, kebab-case)
- Check for file colocation patterns (component + test + style in same dir)
- Check CLAUDE.md for existing conventions

## Step 2: Analyze dependencies

From `package.json` (or equivalent), categorize key dependencies:

- **Core framework** — The primary framework(s)
- **UI libraries** — Component libraries, icon sets, charting
- **State management** — Stores, signals, state machines
- **Data fetching** — API clients, GraphQL, WebSocket libs
- **Auth** — Authentication/authorization libraries
- **Utilities** — Date handling, validation, formatting
- **Dev tooling** — Build tools, bundlers, transpilers

Flag notable patterns:
- Very old versions of core dependencies
- Multiple competing libraries for the same purpose
- Deprecated packages
